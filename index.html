INST309 — Research Context
About This Project
This dashboard was built as part of a semester-long research project exploring how job market knowledge gets lost in traditional data systems — and what it would take to capture it properly.
The Problem We Were Solving
Why this platform exists

Job market data is fragmented across platforms, static in most dashboards, and structured around what is easy to store rather than what is useful to know. The same job gets posted under different titles. Skills that are becoming critical don't surface in time. The context behind a role — how it is shifting, what it is evolving into — gets lost almost entirely in standard database structures.

The goal of this research was to understand what kinds of data capture that invisible knowledge, and what kinds of visualizations make it legible to the people who need to act on it.

The Three Research Roles
How the team divided the problem
🏗 Information Architects

Audited existing SQL schemas, identified semantic gaps, and designed the Neo4j knowledge graph. Found that traditional schemas store stable titles but lose emerging terms, overlap, shifts, and dark data.

🔍 Ethnographers

Searched Reddit, Discord, and forums to find ghost roles and raw practitioner language. Mapped how roles shift from 2020 to 2026 and identified "dark data" — context that exists in practice but never gets formally recorded.

🎨 Product Strategists / UX

Defined the visualization layer. Researched chart types by data type, built the component library, and designed how the data gets shown to users. This dashboard is that deliverable.

Research Timeline
What happened across the 12-week project
Weeks 1–3
Discovery. Reviewed existing job dashboards, identified good and bad UI patterns, read the Syracuse iSchool data type taxonomy, and started classifying which data types appear in job market data.
Weeks 4–6
Landscape Audit. Cross-referenced findings with the Architects' schema analysis and Ethnographers' field notes. The Architects' semantic gap analysis showing what traditional databases miss shaped the chart choices significantly. Seeing the specific types of dark data they documented helped narrow the visualization library to what actually matters.
Weeks 7–9
Blueprinting. Built the data type → visualization mapping. Prototyped each component. Resolved the title fragmentation problem through the Role Similarity component — showing similarity scores based on shared required skills.
Weeks 10–12
Synthesis. Finalized the dashboard prototype, wrote the case study, and prepared this handoff. The dashboard is live at dabarjel.github.io/Job-Market-Research-Dashboard/.
Questions We Resolved
Problems that came up and got answered
How do you handle title fragmentation in a dashboard? → Role Similarity component showing % overlap based on shared required skills. Makes the invisible alias problem visible to the user.
What chart works for showing role demand over time vs. at a point in time? → Line + Threshold for velocity, Ranked List + Movement for the current snapshot. Both are needed.
How do you communicate the difference between rising, new, and shifting roles? → Four distinct card states: Fastest Growing, New Role, Rising Fast, and Name Shifting — each with its own visual treatment and definition.
Questions Still Open
What the next team should tackle first
Who is the primary user — job seeker, recruiter, or labor market researcher? The answer changes the entire information hierarchy.
How do we define a "new" role? First appearance ever, or first time it crosses some posting volume threshold?
What do we do with declining roles? Show them, hide them, flag them?
How does the UX layer connect to the Architects' actual Neo4j schema? The components currently use placeholder data.
